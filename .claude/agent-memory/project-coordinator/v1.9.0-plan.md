# v1.9.0 — Platform Foundations: Implementation Plan

**Theme**: Prepare for the platform transformation
**Features**: Multi-workspace support, Workspace-level settings, Remote plugin registry foundation, Plugin signing system
**Status**: Planning
**Date**: 2026-02-14

---

## Current State Analysis

### What exists today
- **Projects**: Flat in idb-keyval (`project_<id>` keys), no workspace grouping. `ProjectService` (singleton) handles CRUD, current project tracking. `ProjectSettings` has model/ratio/duration/autosave/theme.
- **Settings**: Global `useSettingsStore` (Zustand + persist via idbStorage). Settings include autoSave, apiKey, UI prefs, export, performance. No scoping — all global.
- **Plugins**: `PluginService` (singleton, implements `PluginRegistry`). Supports manifest validation, semver compat, health tracking, crash isolation, permissions. 3 internal studio plugins. All local — no remote fetch. No signing/verification.
- **Database**: idb-keyval (key-value pairs). `DatabaseService` exists with migration framework but most storage uses idb-keyval directly.
- **Types**: `src/core/types/plugin.ts` (334 lines) — full Plugin/Manifest/Context/API types. `src/core/types/index.ts` (680 lines) — all app types.

### What needs to be built
1. **Workspace** abstraction that groups projects + has its own settings
2. **Workspace-scoped settings** that override global defaults
3. **Remote plugin registry** client + types + cache for fetching plugin metadata from a remote source
4. **Plugin signing** — Ed25519 signature verification for plugin integrity

## Architectural Decisions Required

### ADR-011: Workspace Model
**Decision**: A Workspace is a named container of projects with its own settings. All existing projects belong to a `"default"` workspace created on migration. Workspace ID prefixes idb-keyval keys for data isolation.

**Rationale**: Minimal change to existing storage. No IndexedDB schema migration needed — workspace scoping via key prefixes.

### ADR-012: Settings Hierarchy
**Decision**: Settings resolve in order: workspace-level → global-level → hard-coded defaults. Workspace settings are sparse/partial — only overridden fields are stored.

**Rationale**: Matches how VS Code profiles and IDE workspaces handle settings. Existing `useSettingsStore` becomes the "global" layer.

### ADR-013: Remote Plugin Registry Protocol
**Decision**: Registry is a static JSON index hosted on GitHub Pages / CDN. No server-side logic. Client fetches `registry.json` manifest index, each plugin has a metadata entry. Actual plugin bundles are downloaded separately (v2.0.0 scope).

**Rationale**: Static hosting is free, cacheable, and doesn't need authentication infrastructure. Foundation only — actual install from remote deferred to v2.0.0.

### ADR-014: Plugin Signing with Ed25519
**Decision**: Use Web Crypto API (SubtleCrypto) with Ed25519 for plugin manifest + bundle signing. Signatures are base64-encoded and stored in plugin manifest (`signature` field). A public key is bundled with the app. Verification is mandatory for remote plugins, optional for local.

**Rationale**: Ed25519 is fast, small signatures, available via SubtleCrypto. No external crypto library needed. Keeps bundle size small.

## Risks & Mitigations

| # | Risk | Impact | Mitigation |
|---|------|--------|------------|
| 1 | SubtleCrypto Ed25519 not available in jsdom test env | Tests fail | Feature-detect + conditional skip, or use polyfill, or test with ECDSA P-256 fallback |
| 2 | Workspace migration breaks existing project data | Data loss | Idempotent migration in `workspaceService.initialize()`, tested thoroughly |
| 3 | Settings resolution adds latency to renders | UX degradation | Memoize resolved settings, compute once per workspace switch |
| 4 | Registry fetch fails on first use (no server yet) | Confusing UX | Clear "coming soon" messaging, graceful fallback to empty state |
| 5 | Workspace switching with unsaved changes | Data loss | Prompt user to save, or auto-save, before switching workspace |
| 6 | Plugin signing adds friction to plugin dev | Adoption barrier | Soft enforcement in v1.9.0 (warnings only). Provide signing CLI tool. |
